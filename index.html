Riesgos Técnicos Identificados

1. Single Point of Failure (SPOF)

La aplicación monolítica en una sola instancia EC2 significa que cualquier fallo en el servidor detiene toda la operación. Sin redundancia geográfica (Multi-AZ), un problema de hardware, red o incluso mantenimiento programado de AWS puede causar downtime total.

2. Base de Datos No Gestionada

MySQL corriendo localmente en la misma EC2 carece de:
• Backups automáticos con retención configurable
• Failover automático a réplica standby (Multi-AZ)
• Parches de seguridad gestionados por AWS
• Escalado vertical sin downtime significativo

3. Arquitectura Monolítica

Todo el código (UI, lógica de negocio, acceso a datos) vive en un solo codebase. Esto facilita el desarrollo inicial, pero escalar solo una función crítica (ej. procesamiento de rutas) requiere escalar toda la aplicación, desperdiciando recursos y aumentando costos.

Ruta de Migración Recomendada

  1. Fase 1 (Corto Plazo): Migrar MySQL a AWS RDS Multi-AZ para eliminar SPOF de datos y habilitar backups automáticos.
  2. Fase 2 (Mediano Plazo): Implementar Auto Scaling Group con Load Balancer para distribuir tráfico entre múltiples instancias EC2.
  3. Fase 3 (Largo Plazo): Evaluar extracción de módulos críticos (ej. motor de rutas) como microservicios independientes con escalado granular.
Concepto Clave

La arquitectura se refiere a la estructura técnica. La brecha es la falta de escalabilidad y resiliencia: si falla la instancia única, todo cae. Explica la necesidad de migrar a servicios gestionados (RDS, Auto Scaling) para reducir el riesgo de interrupciones.

Análisis de Vulnerabilidades

1. Ausencia de MFA (Multi-Factor Authentication)

Sin MFA, una contraseña comprometida (phishing, keylogger, base de datos filtrada) otorga acceso total al sistema. Estudios demuestran que MFA reduce en 99% los ataques automatizados.

Factores de Autenticación Recomendados:
  • Algo que sabes: Contraseña (actual)
  • Algo que tienes: App móvil (Google Authenticator, Authy) o token físico (YubiKey)
  • Algo que eres: Biometría (huella, reconocimiento facial) - opcional
2. Falta de Audit Logs (Registros de Auditoría)

Sin logs, es imposible responder preguntas críticas: ¿Quién modificó este registro? ¿Cuándo se eliminó esta información? ¿Desde qué IP se accedió? Los audit logs son obligatorios para cumplimiento normativo (ISO 27001, SOC2, HIPAA).

Eventos Críticos a Registrar:
  • Intentos de login (exitosos y fallidos) con timestamp, IP y ubicación
  • Cambios en datos sensibles (trazabilidad, clientes, rutas)
  • Creación/eliminación de usuarios y cambios de permisos
  • Acceso a módulos administrativos o reportes financieros
3. Gestión Reactiva de Usuarios

La desactivación manual de usuarios al salir de la empresa crea una ventana de riesgo. Un ex-empleado puede mantener acceso días o semanas si no se comunica el cambio. Solución: Integración con directorio activo (LDAP/Azure AD) para desactivación automática.

Cumplimiento Normativo

Para certificaciones como ISO 27001 o auditorías SOC2, se requiere evidencia de:
• MFA habilitado para usuarios con acceso a datos sensibles
• Logs de auditoría con retención mínima de 90 días (recomendado: 1 año)
• Revisión periódica de accesos y permisos (trimestral)

Concepto Clave

Se refiere a controles débiles y reactivos. La brecha es la exposición a accesos no autorizados. Explica la necesidad de MFA (Autenticación Multifactor) y auditoría para proteger datos sensibles como la trazabilidad.

Conclusión simple

Cualquiera que entra al sistema tiene usuario y contraseña normal. Cuando alguien deja la empresa, le quitan el acceso solo si alguien avisa. No hay doble verificación (como código en el celular) ni registro de quién hizo qué. Es una puerta sin cerradura fuerte.

Gobernanza de Datos

El Problema de los Silos de Datos

Estado Actual: Fragmentación de la Verdad

Hoy, la misma información (ej. "¿Cuántos kg de aceite recolectamos en enero?") puede tener 3 respuestas diferentes dependiendo de quién la calcule:

  • Logística: Suma manual de tickets en Excel
  • Finanzas: Registros de facturación en otro Excel
  • Sistema Core: Base de datos con registros parciales

Esto crea conflictos en reuniones, decisiones basadas en datos incorrectos, y pérdida de confianza en los reportes.

¿Qué es un SSOT (Single Source of Truth)?

Un SSOT es una base de datos centralizada que actúa como la única fuente de verdad para toda la organización. Todos los sistemas (logística, finanzas, operaciones) consultan y actualizan esta misma base de datos, eliminando discrepancias.

Características de un SSOT:
  • Integración: Todos los datos estructurados viven en un solo lugar
  • Consistencia: La misma consulta siempre devuelve el mismo resultado
  • Accesibilidad: Todos los departamentos acceden a la misma información
  • Trazabilidad: Cada cambio queda registrado con timestamp y usuario
Impacto en Business Intelligence (BI)

Sin un SSOT, herramientas como Power BI o Tableau no pueden funcionar correctamente. Si cada departamento alimenta el BI con sus propios Excels, los dashboards mostrarán datos contradictorios.

Con SSOT: Power BI se conecta directamente a la base de datos central, y todos ven los mismos KPIs actualizados en tiempo real.

El Problema de Excel como Sistema de Registro

❌ Riesgos de Excel:
  • Fórmulas rotas al copiar/pegar
  • Versiones desactualizadas circulando
  • Sin control de acceso (cualquiera edita)
  • No escala con volumen de datos
✅ Ventajas de Base de Datos:
  • Integridad referencial (no datos huérfanos)
  • Transacciones atómicas (todo o nada)
  • Permisos granulares por usuario/rol
  • Consultas optimizadas para millones de registros
Caminos Posibles para Solucionar el Problema de Datos

Para implementar un SSOT (fuente única de verdad) y centralizar los datos, tienes dos caminos:

🔨 Camino A: Reactivar Laravel
  • Tiempo: 6-9 meses hasta tener datos fluyendo
  • Requiere: Auditoría, documentación, equipo técnico dedicado
  • Riesgo: Alto (código sin documentar, sin validación en producción)
  • Costo: CAPEX significativo + OPEX de equipo técnico permanente

Resultado: Sistema a medida, pero con 6-9 meses de espera antes de tener datos centralizados.

🚀 Camino B: Software Comercial
  • Tiempo: 2-3 meses hasta tener datos fluyendo
  • Requiere: Selección de proveedor, configuración, capacitación
  • Riesgo: Bajo (software probado por cientos de empresas)
  • Costo: OPEX mensual predecible, sin sorpresas

Resultado: Datos centralizados en 2-3 meses, listos para analítica y reportes inmediatamente.

Independientemente del camino elegido, el objetivo final es el mismo: tener un SSOT (base de datos centralizada) con información confiable y actualizada para toma de decisiones. La diferencia está en cuánto tiempo tardas en llegar ahí y cuánto riesgo asumes en el camino.

Conclusión simple: Datos y SSOT

La forma en que se guarda y calcula la información importante (cómo se registran recolecciones, pagos, rutas) no está escrita en ningún lado. Solo la saben unas pocas personas. Si esa persona se va, nadie sabe cómo funciona realmente. Es como tener una receta secreta que solo una persona conoce.

Conclusión simple: Analítica

Todas las decisiones importantes (cuánto pagar, cuántas rutas hacer, cuánto se recolectó) se toman mirando hojas de Excel que cada persona arma a su modo. No hay un lugar central con la información correcta y actualizada al momento. Eso hace que las decisiones sean lentas y con riesgo de equivocarse.

Estrategia de Crecimiento: ERP vs CRM

Para escalar, Grupo Sonne debe diferenciar entre el control de la operación interna y la gestión de la relación con clientes.

ERP

"Gestión de lo que sucede adentro"

  • Control de inventarios y activos.
  • Trazabilidad de recolecciones.
  • Integración contable y nómina.
  • Optimización de costos operativos.
Impacto Sonne

Elimina silos de Excel, conectando básculas con facturación automáticamente.

CRM

"Gestión de lo que sucede afuera"

  • Seguimiento de prospectos y ventas.
  • Fidelización y quejas de clientes.
  • Automatización de contratos.
  • Análisis de rentabilidad por cliente.
Impacto Sonne

Asegura que ninguna ruta se pierda y permite ofrecer reportes de impacto ambiental a los clientes.

Concepto Clave

Mientras que el ERP es el cerebro que organiza los recursos y asegura el cumplimiento (SEMARNAT, básculas), el CRM es el motor de ventas que ayuda a crecer la cartera. Sonne necesita primero un robusto control operativo (ERP) para poder soportar el crecimiento que traería un CRM.

Opción A: Reactivar el Desarrollo a Medida (Laravel)

Aprovechar la inversión ya realizada en el sistema Laravel existente, completándolo y poniéndolo en operación.

¿Por qué NO es viable continuar "como está"?

Aunque el sistema existe, no puede usarse en su estado actual por las siguientes barreras críticas:

  • Barrera de Conocimiento: Sin documentación, nadie sabe qué funcionalidades están implementadas, cómo usarlas, o qué falta por completar. Es como tener un manual en blanco.
  • Barrera de Adopción: Los usuarios no pueden empezar a usar algo que no tiene guías, capacitación ni soporte. El sistema quedaría abandonado nuevamente.
  • Barrera de Mantenimiento: Sin equipo técnico que conozca el código, cualquier error o cambio requerido se convierte en un proyecto de "arqueología de código".
  • Barrera de Confianza: Un sistema sin datos históricos, sin validación en producción y sin usuarios activos no genera confianza para operaciones críticas del negocio.
  • Barrera de Integración: Sin APIs documentadas ni conectores probados, no puede integrarse con GPS, contabilidad u otros sistemas necesarios.

Conclusión: El sistema no puede "continuar" porque nunca comenzó a operar. Reactivarlo requiere el mismo esfuerzo (o más) que implementar una solución comercial, pero con mayor riesgo e incertidumbre.

✅ Ventajas de esta opción:
  • Ya existe una base de código desarrollada (no se parte de cero)
  • Puede adaptarse 100% a procesos específicos del negocio
  • Control total sobre funcionalidades y roadmap de desarrollo
  • No hay dependencia de proveedores externos (vendor lock-in)
  • Tecnología moderna con buena comunidad de soporte (Laravel)
⚠️ Desafíos y costos de esta opción:
  • Auditoría inicial obligatoria: Necesaria para conocer el estado real del código (2-3 semanas, costo de consultoría)
  • Documentación completa: Crear manuales técnicos y de usuario desde cero
  • Equipo dedicado: Contratar/mantener desarrolladores Laravel (2-3 personas mínimo)
  • Infraestructura: Implementar ambientes de prueba, backups, seguridad, monitoreo
  • Tiempo de implementación: 6-9 meses estimados hasta operación completa
  • Riesgo de conocimiento: Dependencia de que el equipo técnico documente y transfiera conocimiento
📋 Lo que implicaría elegir esta ruta:
  1. Fase 1 - Auditoría (2-3 semanas): Contratar expertos Laravel para revisar código, evaluar completitud y estimar esfuerzo restante.
  2. Fase 2 - Documentación (1-2 meses): Crear documentación técnica completa del sistema existente.
  3. Fase 3 - Completar desarrollo (3-5 meses): Implementar funcionalidades faltantes según hallazgos de auditoría.
  4. Fase 4 - Infraestructura (1-2 meses): Implementar QA, backups automáticos, monitoreo, seguridad.
  5. Fase 5 - Capacitación y adopción (1 mes): Entrenar usuarios y migrar operaciones desde Excel.

Inversión estimada: Depende de resultados de auditoría. Tiempo total: 6-9 meses aproximadamente.

Opción B: Implementar Software Comercial Especializado

Adoptar soluciones ya probadas en el mercado (como Samsara, MUTA, SimpliRoute) que están diseñadas específicamente para la industria.

✅ Ventajas de esta opción:
  • Implementación rápida: 1-3 meses hasta operación completa (vs 6-9 meses desarrollo propio)
  • Probado y maduro: Software usado por cientos/miles de empresas similares
  • Sin equipo técnico propio: El proveedor mantiene, actualiza y da soporte
  • Actualizaciones automáticas: Nuevas funcionalidades sin costo adicional de desarrollo
  • Integraciones nativas: APIs ya construidas con GPS, contabilidad, etc.
  • Capacitación incluida: Manuales, videos, soporte técnico del proveedor
⚠️ Desafíos y costos de esta opción:
  • Costo recurrente (OPEX): Renta mensual por usuario/vehículo (vs CAPEX único en desarrollo propio)
  • Adaptación de procesos: El negocio debe ajustarse a cómo funciona el software
  • Personalización limitada: Solo puedes usar las funcionalidades que el proveedor ofrece
  • Dependencia del proveedor: Si cambian precios o descontinúan servicio, afecta operaciones
  • Datos en plataforma externa: Información del negocio reside en servidores del proveedor
  • Contratos de largo plazo: Algunos proveedores requieren compromisos anuales
📋 Lo que implicaría elegir esta ruta:
  1. Fase 1 - Selección (2-4 semanas): Evaluar proveedores (Samsara, MUTA, SimpliRoute), solicitar demos y cotizaciones.
  2. Fase 2 - Contratación (1-2 semanas): Negociar contrato, definir número de licencias y alcance.
  3. Fase 3 - Configuración (2-4 semanas): Parametrizar el sistema según procesos actuales, cargar datos maestros.
  4. Fase 4 - Capacitación (2-3 semanas): Entrenar usuarios en el uso del nuevo sistema.
  5. Fase 5 - Go-Live (1-2 semanas): Migración de operaciones y acompañamiento inicial.

Inversión estimada: Renta mensual según proveedor. Tiempo total: 2-3 meses aproximadamente.

Marco para Tomar la Decisión

No existe una respuesta "correcta" universal. La mejor opción depende de tu situación específica. Aquí hay un marco para ayudarte a decidir:

✅ Considera la Opción A (Laravel) si:
  • Tus procesos son muy únicos y ningún software comercial los cubre
  • Tienes presupuesto para 6-9 meses de desarrollo y equipo técnico
  • La auditoría revela que el sistema está 70%+ completo
  • Prefieres control total sobre el roadmap de funcionalidades
  • Quieres evitar costos recurrentes de renta mensual
✅ Considera la Opción B (Comercial) si:
  • Necesitas resultados rápidos (1-3 meses vs 6-9 meses)
  • Tus procesos pueden adaptarse a mejores prácticas de la industria
  • Prefieres no mantener un equipo técnico interno
  • Valoras tener soporte profesional 24/7 del proveedor
  • Quieres aprovechar integraciones ya construidas (GPS, contabilidad, etc.)
Paso Crítico Antes de Decidir:

Si estás considerando la Opción A (Laravel): Es obligatorio hacer una auditoría técnica primero (2-3 semanas). Sin saber el estado real del código, cualquier decisión sería a ciegas. La auditoría te dirá:
• ¿Qué porcentaje está completo?
• ¿Cuánto costaría terminarlo?
• ¿Está bien construido o necesita refactorización?
• ¿Cuánto tiempo tomaría ponerlo en producción?

Opción Híbrida: También puedes usar ambas estrategias en paralelo: implementar software comercial para necesidades urgentes (GPS, rutas) mientras auditas y decides el futuro del Laravel. Esto te da valor inmediato sin cerrar puertas.

Resumen Estratégico Final

Grupo Sonne se encuentra en un punto de decisión estratégica sobre su infraestructura tecnológica.

La situación actual: Existe un sistema Laravel moderno que representa una inversión ya realizada, pero actualmente no está en uso operativo y carece de documentación. Mientras tanto, las operaciones diarias se ejecutan principalmente en Excel y procesos manuales.

Las opciones disponibles: Tienes dos caminos principales: (1) Reactivar y completar el sistema Laravel existente, o (2) Implementar software comercial especializado. Cada opción tiene ventajas y desafíos específicos que se detallan en este documento.

El factor crítico: Independientemente de la opción que elijas, es fundamental tomar una decisión informada basada en datos reales (auditoría técnica si consideras Laravel) y con una visión clara de los recursos (tiempo, presupuesto, equipo) que estás dispuesto a comprometer.

• Evaluar resultados de auditoría/demos

• Tomar decisión basada en datos reales

• Implementar la solución elegida
